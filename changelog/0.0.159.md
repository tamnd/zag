# 0.0.159 — 2026-04-27

The lzma module. Same development flow as 0.0.158: fixture copied from
goipy, spec written before implementation, PR reviewed before merge.

## What changed

### Added lzma module

The `lzma` module wraps the XZ/LZMA compression family the same way
`bz2` wraps bzip2. If you already know how to use `bz2.compress`, the
lzma API is one short hop away.

One-shot compression and decompression work as expected:

```python
import lzma
data = b"hello lzma world " * 50
compressed = lzma.compress(data)
assert lzma.decompress(compressed) == data
```

`compress` accepts a `check` keyword argument to select the integrity
check algorithm (`CHECK_NONE`, `CHECK_CRC32`, `CHECK_CRC64`,
`CHECK_SHA256`). `decompress` handles multi-stream input the same way
the bz2 module does: concatenated XZ blocks decompress cleanly into
their combined contents.

`is_check_supported` tells you whether the implementation can produce
a given check type. All four standard check types return True.

For incremental work, `LZMACompressor` and `LZMADecompressor` give you
the same streaming interface as their bz2 counterparts:

```python
co = lzma.LZMACompressor()
out = co.compress(b"hello ") + co.compress(b"compressor") + co.flush()

d = lzma.LZMADecompressor()
data = d.decompress(out)
print(d.eof)         # True
print(d.unused_data) # b''
```

`LZMADecompressor.decompress` accepts a `max_length` keyword argument
with the same semantics as `BZ2Decompressor`: at most that many bytes
come back in one call, the rest are buffered for the next call.
`needs_input` tells you whether more input is needed before output
can be produced.

`LZMAFile` wraps an XZ-compressed file with the standard file interface:
`read`, `write`, `readline`, `seek`, `tell`, `peek`, and context manager
support. The `name`, `mode`, `readable`, `writable`, and `closed`
attributes all behave as expected.

Finally, `lzma.open` opens an XZ-compressed file in binary or text mode,
matching the `gzip.open` and `bz2.open` signatures.

### Bug fixes

Two bugs in the existing `lzma_mod.zig` were uncovered by the goipy
fixture and fixed:

- `lzma.decompress` was registered without keyword support, so calling
  `lzma.decompress(data, format=lzma.FORMAT_ALONE)` raised a `TypeError`.
  It is now registered with `regKw`, accepting and ignoring unknown
  keyword arguments.

- `LZMAFile` did not store `name` or `mode` as instance attributes, so
  accessing `f.name` or `f.mode` on an open file raised a `PyException`.
  Both are now written into the instance dict during `__init__`.
